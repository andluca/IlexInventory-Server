# Spec: Ilex Inventory Server

## Summary

Backend for Ilex — an inventory management system built natively for F&B CPG brands. Django 5 + DRF on Python 3.12, **raw parameterized PostgreSQL via psycopg, no ORM**. Plain SQL migrations. Append-only `stock_movements` ledger; current stock is a derived query, never stored. Batch-aware with expiration dates, FEFO routing on sales commit, owner-driven recall + recall report. FIFO cost layers compute true COGS per allocation. Owner isolation enforced at three levels: service, composite FKs `(id, owner_id)`, and `@scoped` query helper — cross-owner access returns 404, never 403. 36 endpoints across 7 apps (auth, catalog, procurement, inventory, sales, financials, health). Agent ("Ask Ilex") is Phase 3 with three modes (Query/Draft/Explain), authenticated via Claude Max OAuth, running against a read-only DB role with allowlisted views, 5s statement timeout, and 1000-row cap. Type-safe contract end-to-end: `drf-spectacular` emits OpenAPI 3.1 → `openapi-typescript` generates client types in [`../../IlexInventory-Client`](../../IlexInventory-Client).

Stack: Django 5.x, DRF, psycopg 3, PostgreSQL 16+, drf-spectacular, pytest + pytest-django (real Postgres, no DB mocks), Docker for local dev.

## 2. Foundation

### 2.1 Framework and Runtime

- Django 5.x with Django REST Framework
- Python 3.12+
- TypeScript-strict equivalent at the API boundary via OpenAPI + DRF serializers
- **No Django ORM** (with one exception per BE-D14: `django.contrib.auth.User` is the only ORM-managed model — auth's password hashing + session machinery are reused; everything else is raw psycopg). CI grep gate fails on `from django.db.models` anywhere in `backend/apps/` **except** `apps/core/auth.py`.
- One API class per operation (no `ViewSet`s) — keeps each operation's serializers and permissions explicit
- pytest + pytest-django for tests, against a real Postgres test database (no DB mocks). The `db_test` library ports the `pre_db` / `post_db` state pattern from `epic-test`; see [`../../backend/apps/core/tests/db_test.py`](../../backend/apps/core/tests/db_test.py).

### 2.2 Database

- PostgreSQL 16+
- Plain `.sql` migration files under [`../../backend/migrations/`](../../backend/migrations/), sliced by domain cluster (`0001_init.sql`, `0002_catalog.sql`, `0003_procurement.sql`, `0004_inventory.sql`, `0005_sales.sql`, `0006_views.sql`, `0007_indexes.sql`)
- UUIDv7 primary keys on every table (D5) — generated by a ~10-line Python helper in [`../../backend/apps/core/ids.py`](../../backend/apps/core/ids.py)
- Money and quantity: `numeric(14, 4)` in DB, `Decimal` in Python — **no floats anywhere on money or quantity paths.** CI grep gate fails on `float(` near money/qty modules.
- Stock is a ledger (D1, D3): append-only `stock_movements` table; current on-hand is `SUM(signed_quantity)`, never a stored column. The take-home brief's "Stock" entity maps to our `batches` (each with unique UUIDv7) plus `stock_movements`.
- Owner isolation: composite FKs `(id, owner_id)` on every owner-scoped reference (D4). Cross-table data fusion bugs fail at FK check on insert, not silently in production.
- Read-side projections via SQL `v_*` views: `v_stock_by_batch`, `v_margin_by_product`, `v_recall_report`, `v_expiring_soon`. Agent's read-only role is granted SELECT on views only; tables are revoked.

### 2.3 Architecture (four layers)

```
APIs (apis.py + serializers)         ← thin: validate, auth, dispatch, serialize
   ↓
Services (services.py)               ← write-side: transactions, ledger writes
Selectors (selectors.py)             ← read-side: composition over views and queries
   ↓
Queries (queries/{aggregate}.py)     ← parameterized SQL wrapped in typed functions
   ↓
Schema (migrations/*.sql + views)    ← DDL + integrity rules + triggers
```

Critical rule: data flows top to bottom only. Full responsibilities, import rules, file locations, and naming conventions in [`../architecture.md`](../architecture.md).

CI gates per the executor's contract:
- All tests pass
- No Django ORM imports anywhere
- No SQL outside `queries/` (`grep -RE "cursor\.execute" backend/apps/*/services.py backend/apps/*/selectors.py backend/apps/*/apis.py` returns nothing)
- No floats in money/qty paths
- Every owner-scoped query function uses `@scoped`

### 2.4 Auth and Authorization

**Direct users (web app):**
- DRF `SessionAuthentication` + cookie session
- Sign-up is in scope: email + password, **simplest possible** — no email verification, no password reset (v1)
- All endpoints require auth except the Auth section and `/health`, `/openapi.json`, `/docs`

**Owner isolation (D4):**
- Service-layer owner injection is the primary line of defense — every owner-scoped service receives `owner_id` from the request session, never from the body
- Composite FK `(id, owner_id)` is the database safety net for bugs the service alone can't catch (cross-table data fusion)
- `@scoped` query helper at [`../../backend/apps/core/owner_scope.py`](../../backend/apps/core/owner_scope.py) injects `owner_id = %(owner_id)s` filter; CI gate fails on owner-scoped queries that bypass it
- Cross-owner access returns **404, never 403** — don't leak existence

**Agent (Phase 3):**
- Authenticated via the user's Claude Max OAuth token
- Server-side proxy injects `{ route, filters, selected_ids }` context
- Query tools execute against a read-only DB role granted SELECT on `v_*` views only — tables are revoked
- 5s statement timeout, 1000-row cap, no direct writes (drafts return to UI for confirmation)

### 2.5 Money and Quantity Precision

- DB: `numeric(14, 4)` for both money and quantity
- Python: `decimal.Decimal`; constructors take strings, never floats
- Wire format: JSON strings (e.g., `"100.0000"`), not numbers — preserves precision through TypeScript on the client
- Quantity stored in base units (g, ml, unit) per [`../product.md`](../product.md). The display layer (FE) converts to kg, L on input/output. **The server never sees kg.**
- Profit margin formula: `(revenue − COGS) / COGS × 100%` (D13) — markup, matches the brief's worked example exactly

### 2.6 API Contract

- URL prefix: `/api/v1/`
- OpenAPI 3.1 generated by `drf-spectacular`; consumed by [`../../IlexInventory-Client`](../../IlexInventory-Client) via `openapi-typescript`. Hand-written API types are forbidden on either side.
- Pagination strategy:
  - **Cursor** for `/sales-orders` list and `/movements` audit (high-write, ordering correctness under concurrent inserts)
  - **Offset** for `/products`, `/purchase-orders`, `/batches` (low-write, dev-friendly)
  - Cursor encoding: UUIDv7 of the last item + `created_at` tiebreaker
- **Idempotency-Key header** required on these endpoints (server caches the response and returns the cached body on retry):

| Endpoint | Why |
|---|---|
| `POST /purchase-orders/{id}/receive` | Creates batches + movements atomically — retry can't double-receive |
| `POST /sales-orders/{id}/commit` | FEFO walk + allocations + sale movements — retry can't double-allocate |
| `POST /batches` | Manual batch creation + initial receipt movement |
| `POST /batches/{id}/movements` (kind=`write_off` only) | Removes stock — retry can't double-write-off |
| `POST /batches/{id}/recall`, `un-recall` | Idempotent by design (D3) but accept the header for client uniformity |
| `POST /sales-orders/{id}/void` | Idempotent by design (D8) but accept the header for client uniformity |

- CSV export: `?format=csv` query param on supporting endpoints (DRF content negotiation), streamed `text/csv`
- Cross-owner access: 404, never 403 (D4)
- 4xx errors return `{ "error": "ErrorCode", "detail"?: "human message", "fields"?: { ... } }`

### 2.7 Type Generation

`drf-spectacular` emits OpenAPI 3.1 from DRF serializers and `@extend_schema` decorators. The FE runs `openapi-typescript` against the running server (or a checked-in spec snapshot) to regenerate `src/api/generated/`. CI on the FE fails if regeneration produces a diff.

### 2.8 Agent (Phase 3 — out of v1 BE MVP)

- Single endpoint `POST /api/v1/agent/chat` (described in §3.8)
- Internal tools (allowlisted-view query, SO draft assembly) live as service-layer functions, not public endpoints
- Read-only DB role: `GRANT SELECT ON v_stock_by_batch, v_margin_by_product, v_recall_report, v_expiring_soon TO agent_readonly`. Tables revoked.
- Statement timeout: `SET LOCAL statement_timeout = '5s'` per agent connection
- Row cap: enforced at the tool wrapper level (`LIMIT 1000`)
- Three modes: A1 Query (read-only, autonomous), A2 Draft (returns object to UI, user confirms — never direct write), A3 Explain (read-only, composes multiple queries)
- Skill files (`apps/agent/skills/*.md`) loaded by intent classification — not in v1 BE scope

## 3. Features

All endpoints are versioned `/api/v1/`. Auth required except where noted. Cross-owner returns 404 (D4).

### 3.1 Auth (`apps/core`)

Uses `django.contrib.auth` + DRF `SessionAuthentication`.

| Method | Route | Auth | Description |
|---|---|---|---|
| POST | `/auth/signup` | None | Create account: email + password. No email verification, no password reset. Returns 200 + session cookie set |
| POST | `/auth/login` | None | Set session cookie. 401 on bad credentials |
| POST | `/auth/logout` | Session | Clear session |
| GET | `/auth/me` | Session | Current user info (consumed by FE app shell, ⌘K, agent context) |

**Sign-up flow (X2):**
1. Client POSTs `{ email, password }` to `/auth/signup`
2. Validate email format + uniqueness; password length ≥ 8
3. Create `auth.User` (email = username), set password (Django's hasher)
4. Log user in (set session cookie)
5. Return `{ user: { id, email, created_at } }`

**Login (X1):** standard Django session login. CSRF token via DRF middleware on subsequent state-changing calls.

**Validation:**
- 400 on missing fields, malformed email, password too short
- 409 on duplicate email
- 401 on bad credentials
- All other endpoints reject 401 if no session

### 3.2 Catalog (`apps/catalog`) — products

| Method | Route | Realizes | Description |
|---|---|---|---|
| GET | `/products` | R1 | List; offset pagination. Search by name/SKU; filter `archived={true,false}`. Includes derived on-hand totals from `v_stock_by_batch` |
| POST | `/products` | F2 | Create (name, SKU unique per owner, description, base_unit ∈ `g`/`ml`/`unit`) |
| GET | `/products/{id}` | R1, R6 | Detail with batches summary |
| PATCH | `/products/{id}` | F2 | Edit `name`, `description` only. SKU locked once first batch exists |
| POST | `/products/{id}/archive` | F2 | Soft delete (sets `archived_at`); 409 if no batches exist (use DELETE) |
| DELETE | `/products/{id}` | F2 | Hard delete; 409 if any batch exists (use archive) |
| POST | `/products/import` | F2, F11 | CSV import. Idempotency-Key required. Body: `multipart/form-data` with `file=<csv>`; columns: `name, sku, description, base_unit` |

**Validation:**
- `base_unit` must be one of `g`, `ml`, `unit`
- SKU mutability: once any batch references the product, `PATCH` on `sku` is rejected with 409
- CSV import: rows with malformed `base_unit` or duplicate `sku` are rejected; the response details the failed row indices; partial success is allowed (committed rows persist)

### 3.3 Procurement (`apps/procurement`) — purchase orders

| Method | Route | Realizes | Description |
|---|---|---|---|
| GET | `/purchase-orders` | R5 | List; offset pagination. Filter by status (`draft`/`received`), supplier search, date range |
| POST | `/purchase-orders` | F3 | Create draft. Body: `{ supplier_name, supplier_contact?, lines: [{ product_id, quantity, unit_cost }] }` |
| GET | `/purchase-orders/{id}` | R5 | Detail (lines + post-receive batches) |
| PATCH | `/purchase-orders/{id}` | F3 | Edit draft (replace-style: full lines payload). 409 post-receive (D6) |
| DELETE | `/purchase-orders/{id}` | F3 | Delete draft. 409 post-receive |
| POST | `/purchase-orders/{id}/receive` | F3 | Terminal: creates one batch per line + receipt movements, atomic. Idempotency-Key required. Body: `{ lines: [{ line_id, batch_code, expiration_date? }] }` (owner provides supplier-stamped lot codes; D2 says manual entries skip PO tables, but PO receives fill `purchase_order_line_id` here) |

**Receive flow (F3):**
1. Validate PO is in `draft`, owned by user
2. Validate every line in the body matches a PO line; `batch_code` non-empty; `expiration_date` is a valid date or NULL
3. In one transaction:
   - For each line: insert one `batches` row with `purchase_order_line_id` set, `unit_cost` from the line, `batch_code` and `expiration_date` from the body
   - For each new batch: insert one `stock_movements` row with `kind='receipt'`, `signed_quantity = +quantity`, `reference_type='purchase_order_line'`, `reference_id=line_id`
   - Update `purchase_orders.status = 'received'`
4. Return updated PO with batches

**Validation:**
- 409 if PO already received
- 400 if any line is missing `batch_code` or has an invalid `expiration_date`
- 422 if a line references a product that no longer exists for the owner

### 3.4 Inventory (`apps/inventory`) — batches, movements, recall

| Method | Route | Realizes | Description |
|---|---|---|---|
| GET | `/batches` | R1, R2 | List; offset pagination. Filter by `product_id`, `is_recalled`, `expiring_within={N}` days. Includes derived on-hand from `v_stock_by_batch` |
| GET | `/batches/{id}` | R1, R6 | Detail (on-hand, recall flag, expiration, source PO line if any) |
| POST | `/batches` | F4 | Create manual batch + initial receipt movement (D2: NULL `purchase_order_line_id`, `reference_type='manual'`). Idempotency-Key required. Body: `{ product_id, batch_code, expiration_date?, unit_cost, initial_quantity }` |
| PATCH | `/batches/{id}` | F12 | Correct typos in `batch_code` or `expiration_date` only. Other fields rejected with 400. Writes a `metadata_correction` movement (qty=0) for audit. Naturally idempotent (no-op when value unchanged) |
| POST | `/batches/{id}/movements` | F5, F6 | Record movement. Body: `{ kind, signed_quantity, notes }`. `kind` ∈ `adjustment` (D7), `write_off`. Idempotency-Key required for `write_off`, not for `adjustment` |
| POST | `/batches/{id}/recall` | F9 | Set `is_recalled=true`; write `recall_block` movement (qty=0). Body: `{ reason }` (non-empty). Idempotent by design (D3) |
| POST | `/batches/{id}/un-recall` | F10 | Reverse recall; write `recall_unblock` movement |
| GET | `/batches/{id}/recall-report` | R7, F11 | Customers who received units from this batch via committed, non-voided SOs (`v_recall_report`). Offset pagination. CSV export via `?format=csv` |
| GET | `/movements` | R6, F11 | Cross-cutting audit; cursor pagination. Filter by `batch_id`, `product_id`, `from`, `to`, `kind`. CSV export via `?format=csv` |

**FEFO eligibility predicate** (used by Sales `commit` and the `/sales-orders/{id}/preview` selector):

```sql
WHERE b.owner_id = :owner_id
  AND b.product_id = :product_id
  AND b.is_recalled = FALSE
  AND (b.expiration_date IS NULL OR b.expiration_date >= CURRENT_DATE)
  AND v.on_hand > 0
ORDER BY b.expiration_date ASC NULLS LAST, b.created_at ASC
```

Eligible batches are walked in this order; expired and recalled batches are invisible to FEFO (D11, D9).

**Recall flow (F9):**
1. Validate batch exists for owner; `reason` non-empty
2. If `is_recalled = true` already: skip writes, return current state (idempotent)
3. In one transaction: `UPDATE batches SET is_recalled=true, recall_reason=:reason, recalled_at=NOW()` + insert `stock_movements` row `kind='recall_block', signed_quantity=0, notes=:reason`
4. Return updated batch

**Recall report (R7):** SELECT from `v_recall_report` filtered by `batch_id` and `voided_at IS NULL` on the source SO (D8). Returns `[{ sale_order_id, customer_name, customer_contact, quantity_received, sale_committed_at }]`.

**Metadata correction (F12):** strict allowlist of `{ batch_code, expiration_date }`. Other body keys cause 400. The audit movement records the change so the batch's history reconstructs correctly.

**Validation:**
- 400 if `batches.signed_quantity` is sent in any request other than the implicit `initial_quantity` on `POST /batches` or `signed_quantity` on `POST .../movements`
- 400 if `kind` in `/movements` is not `adjustment` or `write_off`
- 400 if `adjustment` movement has empty `notes` (D7)
- 422 if `write_off` would result in negative on-hand for the batch

### 3.5 Sales (`apps/sales`) — sales orders

| Method | Route | Realizes | Description |
|---|---|---|---|
| GET | `/sales-orders` | R4 | List; cursor pagination. Filter by status, voided, customer search, date range |
| POST | `/sales-orders` | F7 | Create draft. Body: `{ customer_name, customer_contact?, lines: [{ product_id, quantity, sell_price }] }` |
| GET | `/sales-orders/{id}` | R4 | Detail (lines + post-commit allocations + voided_at) |
| PATCH | `/sales-orders/{id}` | F7 | Edit draft (replace-style). 409 post-commit (D6) |
| DELETE | `/sales-orders/{id}` | F7 | Delete draft. 409 post-commit |
| POST | `/sales-orders/{id}/preview` | F7 | FEFO dry-run: returns proposed allocations without committing. Used by FE for the FEFO preview |
| POST | `/sales-orders/{id}/commit` | F7 | Terminal: walks FEFO; writes allocations + sale movements; immutable. Idempotency-Key required. Body may include `{ allocations: [{ line_id, batch_id, quantity }] }` for admin override (D11) |
| POST | `/sales-orders/{id}/void` | F8 | Reversal movements (`sale_void`, positive qty back) + `voided_at`. Allocations remain immutable. Idempotent by design (D8) |

**Commit flow (F7):**
1. Validate SO is in `draft`, owned by user; lines non-empty
2. If `allocations` body is provided: validate every batch belongs to the owner, every product matches the SO line, `SUM(quantity)` per line equals the line's `quantity`. Skip FEFO walk.
3. Otherwise (FEFO walk per line):
   - Lock eligible batches (`SELECT ... FOR UPDATE OF b`) ordered by `expiration_date ASC NULLS LAST, created_at ASC`
   - Greedy-allocate from earliest-expiring batches until line quantity is satisfied
   - If insufficient on-hand: rollback, return 422 with `{ shortfall: { product_id, required, available } }`
4. In one transaction:
   - INSERT one `sale_allocations` row per (line, batch) pair with `unit_cost = batch.unit_cost`
   - INSERT one `stock_movements` row per allocation: `kind='sale'`, `signed_quantity = -allocated_qty`, `reference_type='sales_order_line'`, `reference_id=line_id`
   - UPDATE `sales_orders.status = 'committed'`, `committed_at = NOW()`
5. Return committed SO with allocations

**Void flow (F8):**
1. Validate SO is `committed` and `voided_at IS NULL`. If `voided_at IS NOT NULL`: idempotent return.
2. In one transaction:
   - For each allocation: INSERT one `stock_movements` row with `kind='sale_void'`, `signed_quantity = +allocated_qty`, `reference_type='sale_allocation'`, `reference_id=allocation_id`
   - UPDATE `sales_orders.voided_at = NOW()`
3. Allocations are NOT updated or deleted (D8) — they remain as historical record
4. The `v_recall_report` view filters `WHERE so.voided_at IS NULL`, so voided SOs disappear from recall reports (per D8: voided units are treated as "never effectively left the warehouse")

**Preview flow (`POST .../preview`):** runs the FEFO walk in a savepoint that rolls back, returning the proposed allocations without writing. The FE renders this in the SO draft UI before the user clicks "Commit".

**Validation:**
- 400 on empty lines, missing `customer_name`, non-positive `quantity` or `sell_price`
- 409 on commit if SO is not in `draft`
- 422 on commit if FEFO can't satisfy the order (shortfall details in body)
- 422 on commit if explicit `allocations` reference batches that aren't eligible (recalled, expired, insufficient on-hand)

### 3.6 Financials (`apps/financials`)

| Method | Route | Realizes | Description |
|---|---|---|---|
| GET | `/financials/dashboard` | R3, F11 | Revenue, COGS, profit, margin totals + top-product breakdown. Params: `from`, `to` (default last 30 days). CSV export via `?format=csv` |
| GET | `/financials/margin` | R3, F11 | Per-product margin detail. Cursor pagination. Same date-range params |

**Profit margin formula** (D13):

```
margin_pct = (revenue − COGS) / COGS × 100%
```

Matches the brief's worked example exactly: `$1,000` revenue − `$100` cost = `$900` profit / `$100` cost = **900%**. Labeled "Profit Margin" in API responses and FE dashboard.

**Source views:**
- `v_margin_by_product` joins `sale_allocations` (committed, non-voided) with `batches.unit_cost` and `sales_order_lines.sell_price` per product
- The Selector composes the view with date-range filtering

**Validation:**
- 400 if `from > to`
- 400 if date range > 1 year (perf cap; deferrable)

### 3.7 Reports & exports (cross-app, F11)

CSV export pattern: `?format=csv` query param on the supporting endpoints listed above. Streamed `text/csv` with header row matching the JSON keys, dates ISO-8601, money/qty as raw decimals. No idempotency.

**Endpoints supporting CSV export:** `/financials/dashboard`, `/financials/margin`, `/movements`, `/batches/{id}/recall-report`.

CSV import is also F11 in catalog (`POST /products/import`). Manual stock and POs CSV import is deferred.

## 4. Validation Gates

Every phase is considered complete when:

```bash
pytest                        # all tests pass
mypy backend/                 # type check passes
ruff check backend/           # lint passes
grep -RE "from django.db.models" backend/apps/ | wc -l   # 0
grep -RE "cursor\.execute" backend/apps/*/services.py backend/apps/*/selectors.py backend/apps/*/apis.py | wc -l   # 0
```

Phase-specific:

| Phase | Validation |
|---|---|
| Project setup | `python manage.py runserver` boots without errors; `/api/v1/health` returns 200 with all checks `ok`; OpenAPI regeneration produces a valid 3.1 schema |

## 6. Decisions

D0–D13 are locked in [`../decisions.md`](../decisions.md). Summary table:

| # | Decision | Rejected | Why |
|---|---|---|---|
| D0 | Procurement: header (`purchase_orders`) + lines (`purchase_order_lines`); sales mirror | Flat single table | POs are multi-product in F&B; flat would force supplier/dates/status to repeat per line |
| D1 | `stock_movements.quantity` is signed (positive = in, negative = out) | Unsigned + CASE on kind | Keeps allowlisted views CASE-free; CHECK constraints per kind plug the integrity gap |
| D2 | Manual stock entries skip PO tables; `batches.purchase_order_line_id` nullable | Synthetic "manual" POs | Would pollute every PO listing/report |
| D3 | Recall: `is_recalled` flag on `batches` + qty=0 audit movements | Separate `batch_events` table | Two audit trails, UNION on batch-history queries |
| D4 | Owner isolation: composite FKs `(id, owner_id)` + denormalized `owner_id`; cross-owner = 404 | Service-only; RLS | Service-only is silent on cross-table data fusion; RLS catches forgotten filters but not cross-table consistency |
| D5 | UUIDv7 primary keys | bigserial; UUIDv4 | bigserial leaks row counts; UUIDv4 has poor B-tree locality |
| D6 | PO/SO lifecycle: two states (draft + terminal); terminal immutable, corrections via reversal | Live-on-creation; explicit void state | Live pollutes ledger; explicit void redundant with `voided_at` |
| D7 | Single `kind='adjustment'`; direction via signed qty, reason via notes | Split kinds (`adjustment_in/out`); reason-as-kind enums | Redundant with D1; conflates structure with content |
| D8 | Allocations immutable post-commit; voids via `kind='sale_void'` reversal + `voided_at` | Versioned allocations; UPDATE on allocations | Versioning overkill v1; UPDATE destroys ledger integrity |
| D9 | Recall blocks future sales only; past sales reported, not reversed | Auto-reverse past allocations on recall | Would rewrite committed SOs without consent |
| D10 | Customer/supplier as text + nullable contact fields on PO/SO | Separate customers/suppliers tables | Out of v1 scope per `product.md` |
| D11 | FEFO ignores expired batches; admin overrides via write-off or explicit allocation list | Sell-expired-with-warning | Too easy to miss in a UI warning; F&B compliance |
| D12 | Backend layering: APIs → Services + Selectors → Queries → Schema | 3-layer with inline SQL; Repository pattern | Inline SQL bloats services; Repository imports DDD vocabulary that doesn't fit Django |
| D13 | Profit margin formula = `(revenue − COGS) / COGS × 100%` (markup) | Textbook gross margin `(revenue − COGS) / revenue` | Brief's worked example uses markup explicitly (900% on $100 cost) |
| D14 | `django.contrib.auth.User` is the **only** ORM-managed model — every business entity stays raw SQL via psycopg | Bespoke users table in raw SQL | Re-implementing password hashing + session machinery is high-risk, no v1 benefit |

Local conventions documented elsewhere:

- API versioning `/api/v1/` — locked here, not yet a numbered decision
- Pagination: cursor for high-write lists, offset for low-write lists — locked here
- Idempotency-Key required on the four mutating terminal endpoints — locked here
- Sign-up: simplest possible, no email verification, no password reset — locked here
- Currency: USD only v1 — locked here
- Time zones: storage UTC, display browser-local — locked here
- Margin formula labeling: API and FE label as "Profit Margin" matching brief wording (D13)

If any of these graduate to load-bearing decisions later, they get D14+ in `decisions.md`.
