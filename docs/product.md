# Ilex Inventory

Project context for the Kaizntree take-home challenge. The original brief is in `CHALLENGE.md`. This file captures what the product is, why we made the decisions we made, and the few constraints that matter — not a spec.

---

## What it is

An inventory management system built natively for F&B CPG brands.

The vertical is the wedge. Horizontal incumbents (NetSuite, Cin7) treat F&B as one of many use cases; Ilex assumes it from the schema up. That's the angle of attack — we're not building a generic inventory app and slapping "for F&B" on the marketing page.

Two real users:

- The buyer / ops manager — dashboards, financial views, recall ops.
- The warehouse coordinator — receiving, fulfillment, possibly on an older monitor in fluorescent light, possibly wearing gloves. Floor mode exists for them.

## What makes it different

Five technical choices that separate Ilex from the median submission for this kind of brief, plus one big feature.

### 1. Stock as a ledger, not a counter

Every change to inventory is an append-only insert into `stock_movements` (receipt, sale, adjustment, write-off, recall block). Current stock is a derived view, never a stored column. Free audit trails, point-in-time queries, corrections without destroying history. It's how real inventory systems work — Square, Shopify, NetSuite all do this.

### 2. FIFO cost layers (true COGS)

Each batch is a cost layer with its original unit cost. Sales consume layers in expiration order (FEFO) and write allocations linking each consumed quantity to the batch it came from. COGS for any sale is the sum of `qty × allocation.unit_cost`, not "sale_price minus last_purchase_price." Margins are honest.

### 3. Batches with expiration (FEFO routing)

Every PO line creates a batch with a code, expiration date, and original cost. Sales draw from batches via FEFO. This unlocks the F&B-specific features:

- "Expiring within N days" widgets
- One-click recall: mark batch contaminated → blocks future sales of that batch → returns the list of every customer who received units from it (the recall report)

### 4. Unit precision

The brief lists kg/g, L/mL, unit. Internal storage is always in base units (g, ml, unit). Display layer converts on input/output. The server never sees kg.

### 5. Type-safe contract end-to-end

OpenAPI generated by drf-spectacular → TypeScript types generated from OpenAPI → TanStack Query with typed clients. No hand-written API types on the frontend.

### Plus: the agent

See "Ask Ilex" below — the actual narrative differentiator.

## Stack

- **Backend:** Django 5.x for HTTP, auth, migrations, admin. **Raw parameterized PostgreSQL via psycopg. No ORM.** Business logic in service modules, not in models or views.
- **Frontend:** React + TypeScript (strict), Mantine for components, Tailwind for one-offs, TanStack Query for fetching.
- **Database:** PostgreSQL 16+. Money as `numeric(14, 4)`, never floats. Quantities as `numeric(14, 4)` in base units.
- **Tests:** pytest + pytest-django. Focus on cost-layer math, FEFO selection, recall traceability, profit computation, auth boundaries.
- **Deploy:** Docker for backend, frontend, Postgres. Production runs on Railway (managed Postgres add-on, `DATABASE_URL` auto-injected).

A few constraints worth being explicit about — these are the ones that get silently violated otherwise:

- No Django ORM. Raw SQL via psycopg. Migrations are plain SQL files.
- No floats for money or quantity. Always `numeric(14, 4)` in DB, `Decimal` in Python.
- No naked SQL in route handlers. All SQL goes through service-layer functions.
- Owner-scoped queries always filter on `owner_id = current_user` via a service helper that injects this — there is no other way to query owner-scoped data.
- No `stock_quantity` column on products or batches. Current stock is always derived from `stock_movements`.
- Cross-owner access returns 404, not 403 (don't leak existence).

## Ask Ilex (the agent)

The narrative differentiator. Three modes:

- **Query (read-only).** "What's expiring next week?" "What's my margin on Cold Brew?" Agent calls tools that execute SQL against allowlisted views, returns inline.
- **Draft (write with approval).** "Create a sales order for Acme Café — 20 cans of Cold Brew." Agent assembles the object, returns it as a draft in the UI; user reviews and confirms. Never a direct write.
- **Explain (the smart one).** "Why did Cold Brew margin drop 8% this month?" Agent walks cost layers and movement history, finds that the cheaper batch ran out on April 14th and you've been selling from a more expensive batch since. This is what no dashboard gives you for free.

**Knowledge repo:** markdown skill files in `apps/agent/skills/` (`cost-layers.md`, `fefo.md`, `recall-procedure.md`, `onboarding.md`) loaded into context based on intent classification of the user message. Same shape as Anthropic Skills.

**Context:** every chat call passes `{ route, filters, selected_ids }` from the React app. The agent knows what the user is looking at without browser automation — the React app already knows.

**Guardrails (real, not prompt-level):**

- Read-only Postgres role for query tools
- Allowlisted views only (`v_stock_by_batch`, `v_margin_by_product`, etc.) — agent cannot `SELECT` arbitrary tables
- Statement timeout (5s), row cap (1000)
- Write actions never direct — always returned as drafts the user confirms in UI

**Onboarding integration:** no dedicated tutorial copilot. The agent appears in empty states. Empty product page → "Want me to import from a CSV or extract from a supplier invoice?" Onboarding is just the agent doing its job, surfaced contextually.

## Flows

Product behaviors documented for the agent (loaded as context) and for reviewers. Flows live here; engineering trade-offs live in [`decisions.md`](decisions.md).

### Recall (v1)

A recall is owner-driven. Suppliers are not users of Ilex — single-tenant per user; suppliers communicate problems out-of-band (email, phone, regulatory notice).

Sequence:

1. Owner learns of contamination externally (supplier email, lab test, regulator).
2. Owner finds the batch in Ilex (by batch code, supplier name, product, expiration date).
3. Clicks "Recall this batch", types a free-text reason ("Listeria — supplier email 2026-05-04").
4. System sets `is_recalled=true`, `recalled_at=now`, writes a `recall_block` audit row in the ledger (qty=0).
5. FEFO immediately stops considering the batch — future sales fall through to the next eligible batch.
6. Owner opens the recall report → every customer that received units, with `customer_name` + `customer_contact` from the SOs.
7. Owner reaches out **out-of-band**. Ilex does not send messages.
8. Owner optionally writes off remaining stock (`write_off` movement zeroing the batch).
9. (Rare) If the recall was a mistake, owner clicks "Un-recall" — flag flips to `false`, audit row (`recall_unblock`).

**Past sales are reported, not reversed.** A recall blocks *future* sales of the batch. If the owner wants to refund or recall units already shipped, they explicitly void the SO — recall and SO void are separate user actions, not coupled.

**Voided SOs are excluded from the recall report.** v1 voids mean "the units never effectively left the warehouse"; those customers shouldn't appear.

## Brand

The product is named after *Ilex paraguariensis* — yerba mate. The cultural reference is real (regional, F&B, agricultural) and the visual identity reflects "agriculture meets technical infrastructure."

| | |
|---|---|
| Background | Charcoal `#121212` |
| Primary accent | Tereré green `#00C16A` (the cold-mate preparation specific to MS / Paraguay) |
| Warning (expiring soon) | Amber |
| Warning (expired / recalled) | Clay red |
| UI font | Inter |
| Data font | JetBrains Mono (SKUs, batch IDs, timestamps, prices) |

**Logo: horizontal lockup.** Stylized mate-leaf icon in tereré green, beside a two-tone wordmark — `ilex` in green, `inventory` in white — both Inter, on charcoal. The mark stands alone for favicon and app icon.

**UX motifs:** dense sortable tables (no big graphics), `⌘K` command palette, keyboard-first navigation. Floor mode toggle in nav for warehouse-floor users — high contrast, larger row heights, larger touch targets.

## Out of scope (v1)

- Multi-warehouse / multi-location
- Multi-currency
- Customer / supplier as separate entities (text fields on PO/SO are enough)
- Receiving by photo / PDF (stretch goal — only if the rest ships)
- Manufacturing / Bill of Materials
- Reorder points / par levels
- Orgs / teams (single-tenant per user)
- Partial receipts (PO arrives 3 of 5 lines — workaround: receive what came, manually adjust the rest, or split the PO)
- Partial returns (customer returns some of what they bought — full SO voids only in v1)
- Auto-reversal of past sales on recall (recall blocks future sales; reversing past sales is an explicit SO void)
